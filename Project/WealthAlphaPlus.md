技术栈：Spring Boot、MySQL、Redis、Sentinel、Jetcache

项目介绍：作为招行“投顾模式转型”的核心底座，将财富Alpha+平台积淀的丰富数据资产和专业投研经验，赋能至手机银行App、财富W+平台、财富陪伴平台、数字零售平台等多个零售营销场景，例如基金透视分析、选基表现、五星之选、持仓透视分析、业绩表现、招行解读、理财标签、特色指标、行业分布、收益归因、收益风险指标、AI小招、招小宝、基金经理获奖、基金获奖等专业化服务场景，全面赋能客户投前-投中-投后全生命周期及招行由产品销售向投顾模式转型。

全市场基金1W+，核心热点接口，**峰值 QPS 突破 2W+，面对**大对象（50KB+）高频读取**与**金融级数据一致性的双重挑战，深度参与架构演进与性能攻坚：

难点1：大对象在高并发下网络与序列化性能崩塌：

挑战：

* 例如基金成立以来净值序列，大小可达50KB，在2W QPS峰值下，若全量走Redis，网络带宽消耗巨大，且JSON序列化/反序列化占用大量CPU；

* Redis频繁读取BigKey易导致单分片阻塞，引发集群抖动；

解决：

* 构建“本地（Caffeine）+	分布式（REDIS）”二级缓存架构：
    * 本地缓存拦截95%+读请求，彻底消除网络IO与序列化开销；
    * 通过二级缓存将2W QPS的流量分散到应用节点本地内存，避免Redis单分片因BigKey过载；
* * 数据结构极致压缩
    - 废弃通用 JSON 存储，采用Protobuf进行二进制序列化，去除字段名冗余；
    - 采用设计**自定义二进制协议**：将日期转为 `int` 时间戳，净值放大 10000 倍转为 `long` 数组。
    -
    - **成果**：数据体积压缩 **80%**（从 15KB 降至 3KB），进一步降低 Redis 带宽压力，提升本地缓存命中率。

2、海量数据下的本地缓存内存不可控与碎片化：

挑战：

1、全量数据巨大，若将全市场基金成立以来数据加载到本地，内存瞬间爆炸，容易引发Full GC甚至OOM；

2、不同基金数据大小差异极大（新基金1KB VS 老基金50KB），传统淘汰策略(LRU)失效；

解决：

基于权重的智能淘汰策略(**Weighted LFU**)

- 基于权重的智能淘汰策略 (Weighted LFU)
    - 利用 Caffeine 的 `weigher` 机制，以**对象实际字节大小**为权重，而非简单的条目数。
    - 设定单机缓存水位线（如 **500MB**），自动淘汰冷数据。

- 长尾流量覆盖
    - 基于**二八定律**，仅保留最热门的 300-500 只基金在本地，覆盖 **95%+** 的线上流量；剩余冷数据自动穿透至 Redis，实现**内存成本与性能收益的最优平衡**。

3、分布式集群下的数据一致性问题：

挑战：引入本地缓存后，如何保证二级缓存一致性；

解决：基于Redis Pub/Sub广播失效机制：数据更新时，发布失效消息到Redis频道，所有节点监听并清除本地JVM缓存;

热点数据：采用“后台线程主动异步刷新”，不依赖被动失效，确保核心数据始终鲜活；



4、极端热点引发雪崩风险

挑战：在市场剧烈波动或新发基金抢购时，单只基金可能形成极端热点，

解决：集成Sentinel，实现流量防护和熔断降级机制，防止单点过热问题；
